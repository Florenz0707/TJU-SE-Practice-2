# Config + Bus 答辩问答清单

## 1. 你到底做了什么？

推荐回答：

> 我主要完成了三件事：第一，搭建了独立的 Config Server；第二，把 `gateway`、`order-service`、`user-service` 这几个服务接入配置中心；第三，引入 RabbitMQ 和 Spring Cloud Bus，在 `config-server`、`order-service`、`user-service` 之间建立配置刷新广播链路。

## 2. 你为什么没有按课件使用远程 Git 配置仓库？

推荐回答：

> 我这里是有意改成 native 模式的。因为当前项目主要在本地和 Docker Compose 环境里联调，native 模式不依赖外网、Git 账号和 webhook，演示更稳定，也更容易复现。它的缺点是没有 Git 历史和远程集中托管能力，所以如果继续往完整课程标准或生产方向做，还是建议切回 Git 模式。

## 3. 你现在这套方案还算不算配置中心？

推荐回答：

> 算。因为配置并不是写死在每个服务内部，而是由独立的 Config Server 统一对外提供。只是当前配置源不是远程 Git，而是外部挂载目录，所以它是 Config Center 的替代实现，不是课件原版实现。

## 4. 为什么 Config Server 没有做成集群？

推荐回答：

> 当前我优先完成的是主链路验证，也就是先把 Config Server 跑起来、把客户端接进来、再把 Bus 刷新链路打通。集群化属于下一步扩展，工作量主要是再起一个 Config Server 实例，并把客户端访问方式改成 discovery-first 或通过负载均衡访问。

## 5. 你为什么在 `bootstrap.yml` 里直接写 `config-server:8888`，而不是走 Eureka 发现？

推荐回答：

> 这是一个启动链路简化。因为 Config Client 本来就是在应用很早期拉配置，如果这一步再强依赖服务发现，会增加启动时序复杂度。在 Docker Compose 单网络环境下，直接通过服务名访问更简单、更稳定，也更容易排错。

## 6. 现在有哪些服务真正接入了 Config Client？

推荐回答：

> 当前接入的是 `gateway`、`order-service`、`user-service`。其他服务暂时还是走各自模块本地配置，所以我不会把它表述成“全量服务已经完成迁移”。

## 7. 为什么不是所有服务都接入配置中心？

推荐回答：

> 我采用的是分批迁移策略。先挑链路最关键、最适合演示的服务接入，先把配置拉取和刷新主链路跑通，再继续推广到其他服务。这样排查问题的成本更低，也更符合课程项目的时间安排。

## 8. Bus 真的接进来了吗，还是只写在文档里？

推荐回答：

> Bus 是真的接进来了。代码里 `config-server`、`order-service`、`user-service` 都有 `spring-cloud-starter-bus-amqp` 依赖，`docker-compose.yml` 也确实部署了 RabbitMQ，同时 Config Server 暴露了 `bus-refresh` 端点，所以这不是纸面设计，而是已经进入工程实现。

## 9. 你为什么选 RabbitMQ？

推荐回答：

> Spring Cloud Bus 常见就是 RabbitMQ 或 Kafka。RabbitMQ 在课程项目里部署成本更低、管理界面直观、配置也更简单，所以更适合作为 Bus 的消息中间件。

## 10. 课件里写的是 `spring-cloud-bus` 和 Rabbit Binder，你为什么用了 `spring-cloud-starter-bus-amqp`？

推荐回答：

> 因为当前项目是 Spring Boot 3.3 和 Spring Cloud 2023 的技术栈，这个版本里更常用的是 starter 形式的 `spring-cloud-starter-bus-amqp`。本质上它还是基于 AMQP 和 RabbitMQ 的 Bus，只是依赖写法更符合新版栈。

## 11. `/actuator/bus-refresh` 和 `/actuator/refresh` 有什么区别？

推荐回答：

> `/actuator/refresh` 更偏向单个应用上下文刷新；`/actuator/bus-refresh` 是通过消息总线把刷新事件广播出去，适合多实例、多服务场景。当前我更关注的是 Bus 广播，所以重点开放的是 `bus-refresh`。

## 12. 你现在修改配置后，所有服务都会自动刷新吗？

推荐回答：

> 不能这么说。当前只有接入 Bus 的服务会参与广播刷新，而且代码里没有完整的 `@RefreshScope` 演示点，所以更准确地讲，是“刷新广播基础设施已经接通”，不是“所有服务、所有配置项都已经实现无感热更新”。

## 13. 你为什么要强调 `@RefreshScope`？

推荐回答：

> 因为 Bus 只负责把“刷新事件”广播出去，但具体某个 Bean 能不能在运行时重新读取配置，还要看这个 Bean 的刷新机制。`@RefreshScope` 就是最常见的实现方式。如果没有这一层，很多通过 `@Value` 注入到普通单例 Bean 的值，不一定能形成一个明显的热更新效果。

## 14. 那你当前实现是不是没有真正完成动态刷新？

推荐回答：

> 更准确地说，是“动态刷新链路完成了基础设施接入，但业务侧的可视化演示点没有完全补齐”。所以这部分我会诚实地定义为“部分完成并做了替代实现”，而不是说自己已经把课件里的最终效果全部做满。

## 15. 你为什么保留 `fail-fast: false`？

推荐回答：

> 这是为了提高系统启动时的容错性。配置中心如果暂时不可用，服务仍然可以用本地配置先启动，避免单点故障直接阻断整个系统联调。这是一个偏工程实践的取舍。

## 16. 这会不会让配置不一致？

推荐回答：

> 会增加“配置中心不可用时退回本地配置”的一致性风险，所以它更适合开发和演示环境。在更严格的生产环境里，可以把 `fail-fast` 打开，或者进一步配合统一配置仓库和发布流程来控制一致性。

## 17. 你为什么说 native + volume mount 也算外部化配置？

推荐回答：

> 因为配置文件不在服务代码内部，而是在独立目录中，由 Config Server 统一读取和暴露。服务本身只负责通过 Config Client 获取配置。从“配置与代码分离”的角度看，这已经属于外部化配置，只是外部源不是 Git，而是宿主机目录。

## 18. gateway 为什么接了 Config Client，却没接 Bus？

推荐回答：

> 这是当前实现覆盖范围上的一个边界。gateway 已经能从 Config Server 拉取配置，但我没有把它继续接入 Bus，因此它不属于当前广播刷新样例链路的一部分。这也正好说明我做的是分步接入，而不是一次性全量铺开。

## 19. 如果让你继续完善，你下一步怎么做？

推荐回答：

> 我会优先补三件事。第一，给 `order-service` 或 `user-service` 加一个安全的 `@RefreshScope` 演示 Bean，确保能现场看到热更新效果；第二，把 gateway 和其他业务服务逐步迁移到 Config Client + Bus；第三，再把 Config Server 扩成双实例，并把 native 模式切换成 Git 模式。

## 20. 如果老师问“这算不算完成作业”，你怎么回答最稳？

推荐回答：

> 如果按课件原版标准来衡量，这部分是部分完成，并且我做了替代实现；如果按核心思想来看，配置中心、外部化配置、消息总线广播刷新这三条主线都已经进了代码和部署链路。我的做法更像是先把核心链路跑通，再逐步补全覆盖面和完整性。

## 21. 如果老师问“你这套方案最大的优点是什么？”

推荐回答：

> 最大优点是稳定和可复现。它适合当前课程项目的 Docker Compose 本地部署环境，不依赖外部 Git 服务，配置修改和部署排查都比较直接，能比较稳地把 Config + Bus 这条主链路跑起来。

## 22. 如果老师问“你这套方案最大的缺点是什么？”

推荐回答：

> 最大缺点是还不够完整，主要表现在三点：没有 Git 版本化配置仓库，没有 Config Server 集群，以及没有专门的 `@RefreshScope` 热刷新演示点。所以它更适合作为一个已经落地的替代实现，而不是最终完整版。
