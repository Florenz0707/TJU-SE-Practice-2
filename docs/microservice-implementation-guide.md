# 微服务实现总说明

本文档面向课程验收、组内交接和后续维护，目标不是简单罗列服务名称，而是完整说明这个项目到底做了什么、为什么这样做、每个微服务和集群节点分别承担什么职责，以及一条真实业务请求如何在整套 Spring Cloud 架构里流转。

## 1. 项目现在是什么形态

当前仓库采用的是“前端 + 聚合层 + 多微服务 + Spring Cloud 治理层”的结构，而不是旧版单体。

整体可以分成四层：

1. 前端层：`elm-frontend`
2. 对外聚合层：`elm-v2.0`
3. 业务微服务层：`points-service`、`account-service`、`business-service`、`food-service`、`cart-service`、`order-service`、`address-service`、`user-service`
4. 云治理层：`config-server`、`discovery-server-a`、`discovery-server-b`、`gateway-service`

这套架构不是把原来单体代码复制几份后改端口，而是把不同业务域拆成了独立服务，再补上配置中心、注册中心、网关、服务发现、负载均衡、内部鉴权和容错能力，使其满足课程验收会重点检查的“微服务高层能力”。

## 2. 为什么保留 `elm-v2.0`

`elm-v2.0` 不是一个“还没拆完的遗留包袱”，而是有意保留的聚合层。

保留它的原因有三点：

1. 原课程接口和前端调用路径很多，直接让前端同时面向 8 个服务会显著提高改造成本和演示复杂度。
2. 聚合层能统一处理 JWT、权限、兼容旧接口结构、跨服务编排与补偿，避免把这些横切逻辑散落到每个微服务。
3. 这符合真实工程里的渐进式拆分路径：先让域能力下沉到微服务，再让对外入口保持稳定。

因此，项目现在的对外入口并不是每个微服务直接暴露给前端，而是优先经过 `gateway-service` 和 `elm-v2.0`。

## 3. 每个服务到底负责什么

### 3.1 云治理层

#### `config-server`

- 角色：配置中心
- 作用：统一管理各服务的配置，避免每个服务自己维护一套独立配置
- 当前实现：采用 Spring Cloud Config Server 的 `native` 模式，配置源来自仓库中的本地配置目录
- 为什么是 `native`：课程验收环境更稳定、离线也能跑、少一个 Git 仓依赖，但仍然保留“集中配置管理”的核心能力

#### `discovery-server-a` / `discovery-server-b`

- 角色：注册中心集群
- 技术：Eureka
- 作用：服务启动后向 Eureka 注册，调用方通过服务名而不是固定地址访问下游
- 为什么做双节点：说明系统考虑了注册中心高可用，不是单点注册中心

#### `gateway-service`

- 角色：统一网关
- 作用：
  - 对外暴露统一入口
  - 将 `/api/**` 转发到聚合层 `elm-v2.0`
  - 将 `/services/*/**` 转发到具体微服务，便于验收现场展示“真实微服务入口”
  - 承担服务发现与负载均衡入口
- 当前关键实现：compose 验收场景下已经将上游路由切换为 `lb://service-id`

### 3.2 聚合层

#### `elm-v2.0`

- 角色：对外 API 聚合层
- 主要职责：
  - 保留对前端和课程接口的兼容
  - 统一 JWT 鉴权与当前用户上下文
  - 编排钱包、积分、订单、库存、地址、用户等多个下游服务
  - 做跨服务失败补偿
  - 用 Outbox 处理订单完成/评价后的积分通知

它不是简单的“转发壳”，而是真正承载了跨域编排和业务补偿逻辑。

### 3.3 业务微服务层

#### `points-service`

- 负责积分账户、积分流水、积分规则
- 支持积分冻结、扣减、返还和回滚
- 与订单完成、评价完成链路联动

#### `account-service`

- 负责钱包、交易、优惠券模板与用户券实例
- 对订单下单链路提供扣钱包、核销券、退款、回滚券等能力

#### `business-service`

- 负责商家域数据
- 提供商家快照查询
- 是 `food-service` 校验商家归属的上游服务之一

#### `food-service`

- 负责菜品和库存
- 提供菜品快照查询、库存预占和库存回补
- 当前已使用 OpenFeign 调用 `business-service`
- 容错由 Resilience4j 提供，而不是旧生态里的 Hystrix

#### `cart-service`

- 负责购物车
- 下单前由聚合层读取购物车内容
- 下单成功后由聚合层调用其清空购物车

#### `order-service`

- 负责订单主记录、订单明细、评价
- 订单创建、取消、状态推进、评价创建/删除都在这个域完成

#### `address-service`

- 负责配送地址
- 之所以保持单实例，是因为课程要求里地址服务并不是重点高并发集群目标

#### `user-service`

- 负责用户注册、登录、JWT 签发、用户资料查询
- 当前同样保持单实例，更贴合课程要求中的“用户服务单实例”口径

## 4. 哪些服务做了集群，哪些没有

当前项目在 compose 和本地 cloud 运行口径下，对课程重点服务做了明确区分。

### 4.1 双实例集群服务

这些服务体现“同一服务多实例注册到 Eureka，并通过服务名负载均衡访问”：

- `business-service`: `8083` / `8183`
- `food-service`: `8087` / `8187`
- `cart-service`: `8089` / `8189`
- `order-service`: `8084` / `8184`

### 4.2 单实例服务

- `points-service`: `8081`
- `account-service`: `8082`
- `address-service`: `8085`
- `user-service`: `8086`
- `elm-v2.0`: `8080`
- `gateway-service`: `8090`
- `config-server`: `8888`
- `discovery-server-a`: `8761`
- `discovery-server-b`: `8762` 对外映射，容器内仍是 `8761`

这里最关键的验收点不是“实例数越多越好”，而是：

1. 核心业务服务确实存在双实例注册
2. 网关和客户端确实按服务名访问
3. 下游调用不再写死某个容器地址

## 5. 请求是怎么流转的

### 5.1 用户侧典型链路：下单

一条下单请求的典型流转如下：

1. 前端请求先到 `gateway-service`
2. `gateway-service` 将 `/api/orders` 转发到 `elm-v2.0`
3. `elm-v2.0` 校验当前用户、地址、购物车和商家
4. `elm-v2.0 -> business-service` 查询商家快照
5. `elm-v2.0 -> food-service` 查询菜品并预占库存
6. `elm-v2.0 -> account-service` 核销优惠券、扣减钱包
7. `elm-v2.0 -> points-service` 冻结或扣减积分
8. `elm-v2.0 -> order-service` 创建订单与订单明细
9. `elm-v2.0 -> cart-service` 清理购物车
10. 成功后将订单结果返回前端

这说明订单创建已经不是一个单体事务，而是典型的跨服务编排。

### 5.2 取消订单链路

取消订单时，`elm-v2.0` 会触发相反方向的补偿：

1. `account-service` 退款钱包、回滚券
2. `points-service` 返还或回滚积分
3. `food-service` 回补库存
4. `order-service` 更新订单状态为取消

### 5.3 完单与评价链路

订单完成和评价提交时，不是直接在主事务里同步发积分，而是：

1. 先由 `order-service` 更新订单或评价数据
2. `elm-v2.0` 写 Outbox 事件
3. Outbox 调度再异步通知 `points-service`

这样做的好处是：即使下游积分服务暂时失败，也不会让主业务直接回滚为失败。

## 6. 网关为什么能证明是真正的服务发现

当前 `gateway-service` 的核心价值，不只是“把请求代理一下”。

它体现微服务能力的关键点在于：

1. 通过 Eureka 获取服务实例列表
2. 通过 `lb://service-id` 访问上游，而不是写死 `http://container-name:port`
3. `/services/address/**`、`/services/order/**` 这类直通路径能直接展示拆分后的服务边界

这意味着老师现场如果检查网关配置，看到的不是传统反向代理，而是接入了 Spring Cloud LoadBalancer 的服务发现式网关。

## 7. 配置中心和配置刷新是怎么实现的

### 7.1 集中配置

各服务在启动时会从 `config-server` 拉配置，而不是各自读取一份散落的本地配置。

这样做的直接价值：

- 统一管理不同服务的环境配置
- 方便 cloud profile 和 local profile 区分
- 便于后续做配置刷新

### 7.2 配置刷新

当前项目没有引入完整的 Spring Cloud Bus + MQ，而是通过 `gateway-service` 提供统一刷新入口：

- 刷新入口：`POST /internal/config/refresh`
- 保护头：`X-Config-Refresh-Token`

刷新结果会聚合返回每个实例是否成功，因此它不是黑盒操作，而是可观测的运维入口。

### 7.3 刷新失败时怎么办

当前设计的恢复思路是实例级的：

1. 先看哪些实例刷新失败
2. 单实例失败就对该实例重试或重启
3. 只在必要时做整服务重启

这和当前的双实例部署设计是配套的，说明这个项目考虑过运维兜底而不是只做成功路径演示。

## 8. 内部调用安全边界是怎么做的

微服务拆开后，一个很容易被老师追问的问题是：内部接口是不是谁都能调。

当前项目的回答是：不能。

内部接口统一通过 `X-Internal-Service-Token` 做服务间鉴权，核心口径如下：

1. `/api/inner/**` 这类接口要求内部 token
2. token 不匹配时直接返回 `401`
3. compose 验收环境中 `INTERNAL_SERVICE_TOKEN` 和 `CONFIG_REFRESH_TOKEN` 必须显式配置
4. 源码主配置里已经移除固定弱默认值，避免“没配也能过”的伪安全状态

本地开发场景下，只在 `application-local.properties` 保留 local profile 的本地兜底值，避免影响验收环境的安全口径。

## 9. 服务间调用和容错是怎么做的

### 9.1 服务间调用

当前项目不是完全依赖一种调用方式，而是按职责选择：

- 聚合层 `elm-v2.0` 中大量跨域编排使用内部 HTTP Client
- `food-service -> business-service` 采用 OpenFeign

这说明服务间调用已经从单体内方法调用转为跨进程服务调用。

### 9.2 容错治理

`food-service` 调 `business-service` 时，容错不是用 Hystrix，而是用 Resilience4j。

原因很直接：

1. Hystrix 已经退出主流维护
2. Resilience4j 更符合当前 Spring Cloud 技术栈
3. 对课程目标来说，本质要求是“存在服务调用 + 熔断/降级能力”，不是必须使用某个过时框架

因此，这属于技术栈升级而不是能力缺失。

## 10. 为什么 Compose 启动链现在更稳定

之前微服务项目一个常见问题是：虽然服务都写进了 compose，但启动先后顺序混乱，冷启动时很容易出现配置中心、注册中心、下游服务还没准备好，上游就先起来然后报错。

当前已经做了这几项收口：

1. `config-server`、`discovery-server-a/b`、`elm-v2`、`gateway-service` 增加健康检查
2. `points/account/business/food/cart/order/address/user` 这些业务服务也补齐了健康检查
3. 关键依赖关系从 `service_started` 收紧到 `service_healthy`
4. 业务服务 Dockerfile 已补 `curl`，让 healthcheck 可以真实执行

这意味着当前 compose 不再只是“把容器排在一起”，而是形成了相对可靠的启动链。

## 11. 这套实现为什么符合课程验收关注点

老师通常不会只看“页面能不能点”，而是会看这些高层能力有没有真的体现出来：

### 11.1 有没有真正拆成微服务

有。核心域已经分到独立服务和独立数据库：订单、购物车、商家、菜品、地址、用户、钱包、积分都已有对应服务边界。

### 11.2 有没有服务发现

有。Eureka 双节点注册中心已经落地，Gateway 和客户端通过服务名访问下游。

### 11.3 有没有负载均衡

有。核心业务服务双实例部署，Gateway 路由使用 `lb://service-id`。

### 11.4 有没有配置中心

有。Config Server 统一托管配置，并有集中刷新入口。

### 11.5 有没有内部安全边界

有。内部 token 已统一收口，主配置不再保留固定弱默认值。

### 11.6 有没有服务调用容错

有。至少 `food-service -> business-service` 已明确采用 Feign + Resilience4j。

### 11.7 有没有考虑启动稳定性和运维

有。Compose 健康检查、`service_healthy` 链、刷新失败恢复文档都已经补齐。

## 12. 当前已经完成的工程收口

截至 2026-04-01，微服务高层验收相关的关键收口已经包括：

1. 完整的 Spring Cloud 运行拓扑落地
2. Gateway `lb://service-id` 路由验证
3. 内部 token 显式配置化
4. 业务服务健康检查补齐
5. 配置刷新聚合入口与恢复说明
6. 关键模块测试补齐与回归通过
7. Mockito 测试运行方式工程化收口，避免 JDK 未来禁用 self-attach 时直接失效
8. Spring Cloud LoadBalancer 的 Caffeine cache 依赖已补齐，本地测试输出中不再出现 `LoadBalancerCaffeineWarnLogger`
9. `gateway-service` 的配置刷新入口已补实例发现失败兜底：服务发现整体失败时返回结构化 `503`，单个服务实例发现失败时继续返回聚合结果
10. `user-service` 与 `account-service` 的 `/api/inner/**` 过滤链测试已补齐，内部 token 缺失、错误与合法放行三类路径均已纳入自动化回归

## 13. 当前还剩的非阻断事项

当前没有阻塞课程验收的核心缺口，但还有几项可以继续优化：

1. 显式添加 Mockito javaagent 后，JVM 仍会给出一条 CDS 相关提示，这不影响测试通过，但可以继续研究是否有更安静的启动参数方案
2. 当前配置刷新仍采用 gateway 聚合广播而不是 Spring Cloud Bus，这对课程验收足够，但如果以后继续工程化，可以再接消息总线做更标准的集中刷新
3. 自动化测试已经覆盖了网关、内部 token 和部分补偿链边界，但仍可以继续把更多内部 controller 过滤链测试和跨服务编排异常分支补得更细

这些都属于“让工程更干净”的优化项，而不是课程验收层面的结构性缺陷。

## 14. 一句话总结

这个项目当前的实现重点，不是把原单体拆成多个端口，而是补齐了一套能自圆其说、能真实运行、也能接受验收追问的微服务体系：

- 业务域已拆分
- 关键服务已集群化
- 有注册中心
- 有配置中心
- 有网关
- 有服务发现和负载均衡
- 有内部鉴权
- 有服务容错
- 有启动稳定性和运维恢复设计

因此，从课程验收视角看，它已经具备“完整微服务实现”的主要特征。
