# 配置刷新失败恢复与人工兜底

## 目标

用于老师现场追问“如果配置刷新失败怎么办”时的标准回答，也可直接作为排障手册使用。

## 1. 触发入口

统一入口：`POST /internal/config/refresh`

请求头：`X-Config-Refresh-Token`

返回结果会包含：

- `targetCount`
- `successCount`
- 每个实例的 `serviceId`
- 每个实例的 `instanceId`
- 每个实例的 `success` / `error`

如果服务发现本身不可用，网关会直接返回结构化 `503`，并带上 `message=service discovery unavailable` 与错误详情，便于先区分“刷新失败”还是“注册发现整体故障”。

## 2. 正常恢复顺序

1. 先看网关返回体里哪些实例刷新失败。
2. 如果失败实例数量少于总实例数，优先保留已成功实例继续工作，不做全局回滚。
3. 对失败实例单独重试一次刷新。
4. 若仍失败，再对失败实例执行重启，让其重新从 Config Server 拉取配置。

## 3. 人工兜底步骤

### 场景 A：单个业务实例刷新失败

处理方式：

1. 在 Eureka 中确认该实例仍然存活。
2. 直接重启该实例。
3. 重启后再次调用统一刷新接口，确认该实例恢复。

影响说明：

- 双实例服务下，单个实例失败不会影响整个服务对外可用。
- 这也是保留多实例部署的意义之一。

### 场景 B：某个服务全部实例刷新失败

处理方式：

1. 先检查 Config Server 是否健康。
2. 再检查该服务 `/actuator/refresh` 是否暴露正常。
3. 逐实例重启该服务。
4. 重启后重新调用统一刷新接口验证结果。

### 场景 C：Config Server 本身异常

处理方式：

1. 优先恢复 `config-server`。
2. 确认 `http://localhost:8888/actuator/health` 为 `UP`。
3. 再按顺序刷新或重启下游服务。

### 场景 D：Gateway 直接返回结构化 `503`

处理方式：

1. 先确认 `gateway-service` 返回体中的 `message` 是否为 `service discovery unavailable`。
2. 如果是，优先检查 Eureka 双节点而不是直接重试刷新。
3. 确认 `http://localhost:8761` 可访问，必要时检查 `discovery-server-a/b` 容器或本地进程状态。
4. 注册中心恢复后，再重新调用统一刷新接口。

## 4. 为什么这样设计是合理的

- 当前项目没有引入额外 MQ，总体架构更轻，适合课程验收演示。
- 刷新失败结果是可观测的，不是黑盒。
- 服务发现层异常和实例刷新异常已经可以在返回体层面区分，不会混成同一种 500。
- 失败恢复粒度是“实例级”，不会因为单实例失败拖垮整套服务。
- 对老师来说，这说明项目不仅能演示成功路径，也考虑了运维兜底。

## 5. 现场一句话回答

- 如果配置刷新失败，我们先看网关返回的实例级结果；单实例失败就重试或重启该实例，多实例服务不会整体不可用，这就是我们把刷新入口和实例结果聚合做出来的原因。
